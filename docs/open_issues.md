# CryptoWatcher — Follow-up задачи и операционные риски

## 1. Назначение

Этот документ не является wishlist.

Он нужен для трёх целей:
- фиксировать реальные follow-up задачи после стабилизации текущего MVP
- держать на виду операционные и архитектурные риски
- давать короткий список того, что нужно перепроверять перед деплоем и после него

## 2. Что проверять перед деплоем

Перед выкладкой нужно отдельно проверить:

1. migration legacy state на реальном persisted state-файле
2. правдивость `/status` в обоих режимах радара
3. отсутствие шумных callback/query ошибок на no-op действиях в настройках
4. отсутствие parse error в help/settings/terms
5. корректный startup в текущем Docker deployment flow
6. отсутствие регресса по polling и отправке алертов
7. корректную работу выбранного `signal_timeframe`
8. корректное использование только закрытых свечей выбранного timeframe
9. корректную работу per-chat порогов сигнала
10. согласованность между settings / status / terms / sample alert

## 3. Текущие follow-up темы

### 1. Server time sync
Бот определяет закрытие свечи по локальному времени сервера.

Риск:
если время на VPS заметно съедет, логика “свеча уже закрыта / ещё формируется” может стать неверной.

Что делать:
- убедиться, что на сервере нормально работает time sync
- при необходимости отдельно рассмотреть использование server time от биржи

### 2. Давление на rate limits в `top` режиме
Top-mode radar делает много запросов к kline endpoint.

Риск:
при дальнейшем масштабировании, добавлении новых timeframe или изменении параметров можно подойти к limit pressure и ловить деградацию или 429.

Что делать:
- контролировать поведение на реальном polling cadence
- при расширении scope отдельно рассмотреть batching / throttling

### 3. Legacy / dead code вне main path
В коде ещё могут оставаться старые функции и ветки, не являющиеся текущим основным продуктовым путём.

Риск:
это повышает стоимость изменений, вероятность регрессов и сложность дальнейшего code review.

Что делать:
- вынести cleanup в отдельный follow-up спринт
- не смешивать большой cleanup с production bugfix deploy

### 4. Недостаток лёгких regression checks
Критичные места уже известны:
- migration
- truthful `/status`
- no-op callback behavior
- `signal_timeframe`
- closed-candle filtering
- per-chat threshold overrides

Риск:
эти участки легко снова сломать в будущих PR.

Что делать:
- добавить лёгкие code-level regression checks без тяжёлой тестовой инфраструктуры
- как минимум сохранить воспроизводимые локальные сценарии проверки

### 5. Большой монолитный файл
`telegram_crypto_watcher.py` остаётся крупным и смешивает:
- Telegram handlers
- state logic
- fetch layer
- signal evaluation
- rendering

Риск:
чем дальше продукт будет расти, тем выше станет цена локальной правки и шанс побочных эффектов.

Что делать:
- рассмотреть modularization отдельным спринтом после стабилизации текущего MVP

### 6. UX-polish терминов, статуса и alert preview
Текущие terms/settings/status уже приведены к более понятному русскому UX и per-chat порогам, но остаётся пространство для улучшений.

Риск:
бот остаётся технически корректным, но визуально и смыслово объясняет логику слабее, чем мог бы.

Что делать:
- сохранять единые пользовательские названия (`Движение цены`, `Спайк объёма`, `Ликвидность 24ч`) во всех новых изменениях
- улучшать status rendering без нарушения Telegram-safe ограничений длины
- развивать preview / sample alert без усложнения основного UX

### 7. Preset strategy for thresholds
Настройки порогов сигнала через бот добавляют больше UX-гибкости, но могут породить слишком чувствительные или слишком “немые” конфигурации.

Риск:
пользователь меняет пороги и потом не понимает, почему сигналов слишком много или слишком мало.

Что делать:
- использовать безопасные preset-значения
- явно показывать активные значения в settings/status
- не вводить сложный free-text ввод без сильной причины

## 4. Уже известные исторические регрессии

В предыдущих итерациях уже встречались:

- parse error в help/settings тексте
- `Message is not modified`
- шумный CMC fallback log spam
- неконсистентность `/status` относительно реального alert universe
- проблемы migration при изменении state shape
- риск слишком длинного `/status`

Эти зоны нужно считать чувствительными и проверять повторно после изменений.

## 5. Что не делать в follow-up без явной причины

Нельзя без отдельного продуктового решения:

- возвращать CMC как основной путь
- возвращать старую baseline-driven модель как main product path
- переносить управление пользовательским списком обратно в `.env`
- смешивать безопасный deploy-fix с большим cleanup/refactor PR
- превращать настройки порогов сигнала в сложный конструктор с произвольным вводом без необходимости
